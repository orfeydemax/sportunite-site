# Создатель навыков (Skill Creator)

Этот навык предоставляет руководство по созданию эффективных навыков.

## О навыках

Навыки — это модульные, самодостаточные пакеты, которые расширяют возможности Claude, предоставляя
специализированные знания, рабочие процессы и инструменты. Думайте о них как о «руководствах по адаптации» для конкретных
областей или задач — они превращают Claude из агента общего назначения в специализированного агента,
обладающего процедурными знаниями, которыми не может полностью обладать ни одна модель.

### Что предоставляют навыки

1. Специализированные рабочие процессы - Многоэтапные процедуры для конкретных областей.
2. Интеграция инструментов - Инструкции по работе с конкретными форматами файлов или API.
3. Экспертиза в предметной области - Знания о компании, схемы, бизнес-логика.
4. Пакетные ресурсы - Скрипты, ссылки и ресурсы для сложных и повторяющихся задач.

## Основные принципы

### Лаконичность — это ключ

Окно контекста — это общее благо. Навыки делят окно контекста со всем остальным, что нужно Claude: системным промптом, историей разговоров, метаданными других навыков и фактическим запросом пользователя.

**Предположение по умолчанию: Claude уже очень умен.** Добавляйте только тот контекст, которого у Claude еще нет. Проверяйте каждую часть информации вопросами: «Действительно ли Claude нужно это объяснение?» и «Оправдывает ли этот абзац свою стоимость в токенах?»

Отдавайте предпочтение кратким примерам, а не многословным объяснениям.

### Установка соответствующих степеней свободы

Соответствуйте уровню специфичности хрупкости и изменчивости задачи:

**Высокая свобода (текстовые инструкции)**: Используйте, когда допустимы несколько подходов, решения зависят от контекста или подход направляется эвристикой.

**Средняя свобода (псевдокод или скрипты с параметрами)**: Используйте, когда существует предпочтительный шаблон, допустимы некоторые вариации или конфигурация влияет на поведение.

**Низкая свобода (конкретные скрипты, минимум параметров)**: Используйте, когда операции хрупки и склонны к ошибкам, критична последовательность или должна соблюдаться определенная последовательность.

Думайте о Claude как об исследователе пути: узкий мост над обрывом требует конкретных перил (низкая свобода), в то время как открытое поле допускает множество маршрутов (высокая свобода).

### Анатомия навыка

Каждый навык состоит из обязательного файла SKILL.md и дополнительных пакетных ресурсов:

```
skill-name/
├── SKILL.md (обязательно)
│   ├── Метаданные YAML frontmatter (обязательно)
│   │   ├── name: (обязательно)
│   │   ├── description: (обязательно)
│   │   └── compatibility: (опционально, редко нужно)
│   └── Инструкции Markdown (обязательно)
└── Пакетные ресурсы (опционально)
    ├── scripts/          - Исполняемый код (Python/Bash и т.д.)
    ├── references/       - Документация, предназначенная для загрузки в контекст по мере необходимости
    └── assets/           - Файлы, используемые в выводе (шаблоны, иконки, шрифты и т.д.)
```

#### SKILL.md (обязательно)

Каждый SKILL.md состоит из:

- **Frontmatter** (YAML): Содержит поля `name` и `description` (обязательно), а также дополнительные поля, такие как `license`, `metadata` и `compatibility`. Только `name` и `description` читаются Claude для определения того, когда срабатывает навык, поэтому будьте ясны и исчерпывающи в том, что представляет собой навык и когда его следует использовать. Поле `compatibility` предназначено для указания требований среды (целевой продукт, системные пакеты и т.д.), но большинству навыков оно не требуется.
- **Body** (Markdown): Инструкции и рекомендации по использованию навыка. Загружаются только ПОСЛЕ срабатывания навыка (если загружаются вообще).

#### Пакетные ресурсы (опционально)

##### Скрипты (`scripts/`)

Исполняемый код (Python/Bash и т.д.) для задач, требующих детерминированной надежности или постоянно переписываемых.

- **Когда включать**: Когда один и тот же код переписывается многократно или требуется детерминированная надежность.
- **Пример**: `scripts/rotate_pdf.py` для задач по повороту PDF.
- **Преимущества**: Экономия токенов, детерминированность, возможность выполнения без загрузки в окно контекста.
- **Примечание**: Скрипты все равно могут понадобиться Claude для чтения при внесении исправлений или адаптации под среду.

##### Ссылки (`references/`)

Документация и справочные материалы, предназначенные для загрузки в контекст по мере необходимости для информирования процесса и размышлений Claude.

- **Когда включать**: Для документации, на которую Claude должен ссылаться во время работы.
- **Примеры**: `references/finance.md` для финансовых схем, `references/mnda.md` для шаблона NDA компании, `references/policies.md` для политик компании, `references/api_docs.md` для спецификаций API.
- **Варианты использования**: Схемы баз данных, документация API, знания в предметной области, политики компании, подробные руководства по рабочим процессам.
- **Преимущества**: Сохраняет SKILL.md компактным, загружается только тогда, когда Claude решит, что это необходимо.
- **Лучшая практика**: Если файлы большие (>10 тыс. слов), включите шаблоны поиска grep в SKILL.md.
- **Избегайте дублирования**: Информация должна находиться либо в SKILL.md, либо в справочных файлах, но не в обоих местах. Отдавайте предпочтение справочным файлам для подробной информации, если только она не является действительно ключевой для навыка — это сохраняет SKILL.md компактным, делая информацию доступной без переполнения окна контекста. Оставляйте в SKILL.md только основные процедурные инструкции и руководство по рабочему процессу; перемещайте подробные справочные материалы, схемы и примеры в справочные файлы.

##### Ассеты (`assets/`)

Файлы, не предназначенные для загрузки в контекст, а используемые в продукте, который производит Claude.

- **Когда включать**: Когда навыку нужны файлы, которые будут использоваться в конечном результате.
- **Примеры**: `assets/logo.png` для брендированных ассетов, `assets/slides.pptx` для шаблонов PowerPoint, `assets/frontend-template/` для заготовок HTML/React, `assets/font.ttf` для типографии.
- **Варианты использования**: Шаблоны, изображения, иконки, шаблонный код, шрифты, образцы документов, которые копируются или изменяются.
- **Преимущества**: Отделяет выходные ресурсы от документации, позволяет Claude использовать файлы без их загрузки в контекст.

#### Что не следует включать в навык

Навык должен содержать только основные файлы, которые напрямую поддерживают его функциональность. НЕ создавайте постороннюю документацию или вспомогательные файлы, включая:

- README.md
- INSTALLATION_GUIDE.md
- QUICK_REFERENCE.md
- CHANGELOG.md
- и т.д.

Навык должен содержать только ту информацию, которая необходима ИИ-агенту для выполнения поставленной задачи. Он не должен содержать вспомогательный контекст о процессе его создания, процедурах настройки и тестирования, документации для пользователя и т.д. Создание дополнительных файлов документации только добавляет беспорядка и путаницы.

### Принцип проектирования прогрессивного раскрытия (Progressive Disclosure)

Навыки используют трехуровневую систему загрузки для эффективного управления контекстом:

1. **Метаданные (имя + описание)** - Всегда в контексте (~100 слов).
2. **Тело SKILL.md** - Когда навык срабатывает (<5 тыс. слов).
3. **Пакетные ресурсы** - По мере необходимости Claude (неограниченно, так как скрипты можно выполнять без чтения в окно контекста).

#### Паттерны прогрессивного раскрытия

Ограничивайтесь самым важным в теле SKILL.md и держите его в пределах 500 строк, чтобы минимизировать раздувание контекста. Разделяйте содержимое на отдельные файлы при приближении к этому пределу. При выносе содержимого в другие файлы очень важно ссылаться на них из SKILL.md и четко описывать, когда их нужно читать, чтобы читатель навыка знал об их существовании и о том, когда их использовать.

**Ключевой принцип:** Когда навык поддерживает несколько вариаций, фреймворков или опций, оставляйте в SKILL.md только основной рабочий процесс и руководство по выбору. Переносите специфические детали вариантов (шаблоны, примеры, конфигурацию) в отдельные справочные файлы.

**Паттерн 1: Высокоуровневое руководство со ссылками**

```markdown
# Обработка PDF

## Быстрый старт

Извлечение текста с помощью pdfplumber:
[code example]

## Продвинутые функции

- **Заполнение форм**: См. [FORMS.md](FORMS.md) для полного руководства.
- **Справочник API**: См. [REFERENCE.md](REFERENCE.md) для всех методов.
- **Примеры**: См. [EXAMPLES.md](EXAMPLES.md) для общих шаблонов.
```

Claude загружает FORMS.md, REFERENCE.md или EXAMPLES.md только при необходимости.

**Паттерн 2: Организация по предметным областям**

Для навыков с несколькими областями организуйте содержимое по областям, чтобы избежать загрузки нерелевантного контекста:

```
bigquery-skill/
├── SKILL.md (обзор и навигация)
└── reference/
    ├── finance.md (доход, метрики выставления счетов)
    ├── sales.md (сделки, воронка продаж)
    ├── product.md (использование API, функции)
    └── marketing.md (кампании, атрибуция)
```

Когда пользователь спрашивает о метриках продаж, Claude читает только sales.md.

Аналогично, для навыков, поддерживающих несколько фреймворков или вариантов, организуйте по вариантам:

```
cloud-deploy/
├── SKILL.md (рабочий процесс + выбор провайдера)
└── references/
    ├── aws.md (паттерны развертывания AWS)
    ├── gcp.md (паттерны развертывания GCP)
    └── azure.md (паттерны развертывания Azure)
```

Когда пользователь выбирает AWS, Claude читает только aws.md.

**Паттерн 3: Условные детали**

Показывайте основное содержимое, давайте ссылки на продвинутое:

```markdown
# DOCX Processing

## Создание документов

Используйте docx-js для новых документов. См. [DOCX-JS.md](DOCX-JS.md).

## Редактирование документов

Для простых правок изменяйте XML напрямую.

**Для отслеживания исправлений**: См. [REDLINING.md](REDLINING.md)
**Для деталей OOXML**: См. [OOXML.md](OOXML.md)
```

Claude читает REDLINING.md или OOXML.md только тогда, когда пользователю нужны эти функции.

**Важные указания:**

- **Избегайте глубоко вложенных ссылок** - Ссылки должны быть на один уровень глубже от SKILL.md. Все справочные файлы должны ссылаться напрямую из SKILL.md.
- **Структурируйте длинные справочные файлы** - Для файлов длиннее 100 строк включите оглавление в начале, чтобы Claude мог видеть весь объем при предварительном просмотре.

## Процесс создания навыка

Создание навыка включает следующие этапы:

1. Понимание навыка на конкретных примерах.
2. Планирование содержимого многоразового навыка (скрипты, ссылки, ассеты).
3. Инициализация навыка (запуск init_skill.py).
4. Редактирование навыка (реализация ресурсов и написание SKILL.md).
5. Упаковка навыка (запуск package_skill.py).
6. Итерация на основе реального использования.

Следуйте этим шагам по порядку, пропуская только в том случае, если есть веская причина, почему они неприменимы.

### Шаг 1: Понимание навыка на конкретных примерах

Пропускайте этот шаг только тогда, когда шаблоны использования навыка уже четко понятны. Он остается ценным даже при работе с существующим навыком.

Чтобы создать эффективный навык, четко разберитесь в конкретных примерах того, как он будет использоваться. Это понимание может прийти либо из прямых примеров пользователя, либо из сгенерированных примеров, подтвержденных отзывами пользователей.

Например, при создании навыка редактора изображений (image-editor) актуальными будут вопросы:

- «Какую функциональность должен поддерживать навык редактора изображений? Редактирование, поворот, что-то еще?»
- «Не могли бы вы привести несколько примеров того, как этот навык будет использоваться?»
- «Я могу представить, как пользователи просят что-то вроде "Удали эффект красных глаз на этом снимке" или "Поверни это изображение". Есть ли другие способы использования этого навыка, которые вы представляете?»
- «Что должен сказать пользователь, чтобы сработал этот навык?»

Чтобы не перегружать пользователей, избегайте задавать слишком много вопросов в одном сообщении. Начните с самых важных вопросов и уточняйте по мере необходимости для повышения эффективности.

Завершайте этот шаг, когда появится четкое представление о функциональности, которую должен поддерживать навык.

### Шаг 2: Планирование содержимого многоразового навыка

Чтобы превратить конкретные примеры в эффективный навык, проанализируйте каждый пример:

1. Рассмотрите, как выполнить пример с нуля.
2. Определите, какие скрипты, ссылки и ассеты были бы полезны при многократном выполнении этих рабочих процессов.

Пример: При создании навыка `pdf-editor` для обработки запросов типа «Помоги мне повернуть этот PDF», анализ показывает:

1. Поворот PDF требует каждый раз переписывать один и тот же код.
2. Было бы полезно сохранить скрипт `scripts/rotate_pdf.py` в навыке.

Пример: При разработке навыка `frontend-webapp-builder` для запросов типа «Создай мне приложение todo» или «Создай мне дашборд для отслеживания моих шагов», анализ показывает:

1. Написание фронтенд-веб-приложения каждый раз требует одного и того же шаблонного кода HTML/React.
2. Было бы полезно сохранить шаблон `assets/hello-world/`, содержащий файлы проекта с шаблонным HTML/React, в навыке.

Пример: При создании навыка `big-query` для обработки запросов вроде «Сколько пользователей вошло в систему сегодня?», анализ показывает:

1. Запросы к BigQuery требуют каждый раз заново изучать схемы таблиц и связи между ними.
2. Было бы полезно сохранить файл `references/schema.md` с документированными схемами таблиц в навыке.

Чтобы сформировать содержимое навыка, проанализируйте каждый конкретный пример и составьте список многоразовых ресурсов: скриптов, ссылок и ассетов.

### Шаг 3: Инициализация навыка

На этом этапе пора приступить к фактическому созданию навыка.

Пропускайте этот шаг только в том случае, если разрабатываемый навык уже существует и требуется итерация или упаковка. В этом случае переходите к следующему шагу.

При создании нового навыка с нуля всегда запускайте скрипт `init_skill.py`. Скрипт удобно генерирует новый каталог шаблона навыка, который автоматически включает все необходимое, что делает процесс создания навыка гораздо более эффективным и надежным.

Использование:

```bash
scripts/init_skill.py <имя-навыка> --path <выходной-каталог>
```

Скрипт:

- Создает каталог навыка по указанному пути.
- Генерирует шаблон SKILL.md с правильным frontmatter и заглушками TODO.
- Создает примеры каталогов ресурсов: `scripts/`, `references/` и `assets/`.
- Добавляет примеры файлов в каждый каталог, которые можно настроить или удалить.

После инициализации настройте или удалите сгенерированный SKILL.md и примеры файлов по мере необходимости.

### Шаг 4: Редактирование навыка

При редактировании (только что сгенерированного или существующего) навыка помните, что навык создается для использования другим экземпляром Claude. Включайте информацию, которая была бы полезна и неочевидна для Claude. Подумайте, какие процедурные знания, специфические для области детали или многоразовые ассеты помогли бы другому экземпляру Claude выполнять эти задачи более эффективно.

#### Изучите проверенные паттерны проектирования

Обратитесь к этим полезным руководствам в зависимости от потребностей вашего навыка:

- **Многоэтапные процессы**: См. references/workflows.md для последовательных рабочих процессов и условной логики.
- **Специфические форматы вывода или стандарты качества**: См. references/output-patterns.md для шаблонов и примеров паттернов.

Эти файлы содержат устоявшиеся лучшие практики для разработки эффективных навыков.

#### Начните с многоразового содержимого навыка

Чтобы начать реализацию, начните с выявленных выше многоразовых ресурсов: файлов в `scripts/`, `references/` и `assets/`. Обратите внимание, что этот шаг может потребовать ввода данных от пользователя. Например, при реализации навыка `brand-guidelines` пользователю может потребоваться предоставить брендированные ассеты или шаблоны для хранения в `assets/` или документацию для хранения в `references/`.

Добавленные скрипты должны быть протестированы путем их фактического запуска, чтобы убедиться в отсутствии ошибок и соответствии вывода ожидаемому. Если имеется много похожих скриптов, достаточно протестировать репрезентативную выборку, чтобы быть уверенным в их работоспособности, соблюдая баланс со временем завершения.

Любые файлы примеров и каталоги, не нужные для навыка, следует удалить. Скрипт инициализации создает примеры файлов в `scripts/`, `references/` и `assets/` для демонстрации структуры, но большинству навыков они не понадобятся в полном объеме.

#### Обновите SKILL.md

**Рекомендации по написанию:** Всегда используйте повелительную форму или неопределенную форму глагола.

##### Frontmatter

Напишите YAML frontmatter с полями `name` и `description`:

- `name`: Имя навыка.
- `description`: Это основной механизм срабатывания вашего навыка, который помогает Claude понять, когда его использовать.
  - Укажите как то, что делает навык, так и конкретные триггеры/контексты для его использования.
  - Всю информацию о том, «когда использовать», указывайте здесь, а не в теле. Тело загружается только после срабатывания, поэтому разделы «Когда использовать этот навык» в теле бесполезны для Claude.
  - Пример описания для навыка `docx`: «Всестороннее создание, редактирование и анализ документов с поддержкой отслеживания исправлений, комментариев, сохранения форматирования и извлечения текста. Используйте, когда Claude нужно работать с профессиональными документами (файлы .docx) для: (1) создания новых документов, (2) модификации или редактирования контента, (3) работы с отслеживанием исправлений, (4) добавления комментариев или любых других задач по документам».

Не включайте никакие другие поля в YAML frontmatter.

##### Тело (Body)

Напишите инструкции по использованию навыка и его пакетных ресурсов.

### Шаг 5: Упаковка навыка

Как только разработка навыка завершена, его необходимо упаковать в распространяемый файл .skill, который передается пользователю. Процесс упаковки автоматически сначала валидирует навык, чтобы убедиться, что он соответствует всем требованиям:

```bash
scripts/package_skill.py <путь/к/папке-навыка>
```

Опциональное указание выходного каталога:

```bash
scripts/package_skill.py <путь/к/папке-навыка> ./dist
```

Скрипт упаковки:

1. **Валидирует** навык автоматически, проверяя:
   - Формат YAML frontmatter и обязательные поля.
   - Соглашения об именовании навыков и структуру каталогов.
   - Полноту и качество описания.
   - Организацию файлов и ссылки на ресурсы.

2. **Упаковывает** навык, если валидация прошла успешно, создавая файл .skill, названный в честь навыка (например, `my-skill.skill`), который включает все файлы и сохраняет правильную структуру каталогов для распространения. Файл .skill — это zip-архив с расширением .skill.

Если валидация не удалась, скрипт сообщит об ошибках и завершит работу без создания пакета. Исправьте все ошибки валидации и снова запустите команду упаковки.

### Шаг 6: Итерация

После тестирования навыка пользователи могут запросить улучшения. Часто это происходит сразу после использования навыка, на свежем опыте того, как он себя показал.

**Рабочий процесс итерации:**

1. Используйте навык в реальных задачах.
2. Замечайте трудности или неэффективность.
3. Определите, как следует обновить SKILL.md или пакетные ресурсы.
4. Внесите изменения и протестируйте снова.
